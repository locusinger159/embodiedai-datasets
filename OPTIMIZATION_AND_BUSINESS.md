# Part I — 优化方向

## 1. 架构优化

### 1.1 构建器拆分方案

**现状问题**

当前 `build.cjs` 单文件 1206 行，承担了：
- i18n 字典定义（70 行）
- 模板替换引擎（20 行）
- 数据集详情页构建（250 行）
- 标准详情页构建（80 行）
- Benchmark 排行榜构建（100 行）
- 工具详情页构建（60 行）
- 博客构建（50 行）
- 格式全景图、标准草案等（100 行）
- 工具函数（esc / formatDescription / buildSchemaTree / splitOutsideParens，80 行）
- 主流程（100 行）

这导致：修改任一页面都要在一个大文件里定位；i18n 字典与逻辑混在一起；难以单独测试。

**目标拆分结构**

```
build/
├── index.js                  # 主入口，编排构建流程
├── i18n/
│   ├── zh.json               # 中文字典（UI 字符串）
│   └── en.json               # 英文字典
├── utils/
│   ├── esc.js                # HTML 转义
│   ├── format-description.js # 描述文本转 HTML
│   ├── schema-tree.js        # Schema 树渲染
│   └── split-outside-parens.js
├── pages/
│   ├── home.js               # 首页
│   ├── datasets-list.js      # 数据集列表
│   ├── dataset-detail.js     # 数据集详情（含相关推荐算法）
│   ├── standards-list.js
│   ├── standard-detail.js    # 含 RFC 2119 要求渲染
│   ├── tools-list.js
│   ├── tool-detail.js
│   ├── benchmarks.js         # 排行榜索引 + 单页
│   ├── recommend.js          # 反向推荐
│   ├── blog.js
│   ├── formats.js            # 格式全景图
│   ├── standard-proposal.js
│   └── submit.js
└── templates/
    └── render.js             # 通用模板替换引擎
```

**迁移策略**

不要一次性重写。采用「**绞杀者模式（Strangler Fig）**」：

1. 先抽 `utils/`（纯函数，无副作用），`build.cjs` 改为 `require('./build/utils/esc')`；
2. 再抽 `i18n/`（JSON 数据），验证中英文构建结果一致；
3. 逐个抽 `pages/`，每抽一个跑一次完整构建对比 `dist/` diff，确保零变化；
4. 最后 `build.cjs` 只剩主流程编排，可重命名为 `build/index.js`。

**预期收益**

- 单文件最大 200 行，可读性提升；
- i18n 字典外置后，非开发者可直接编辑 JSON 贡献翻译；
- 每个页面构建逻辑可单独单元测试（用 `node:test`，零依赖）。

### 1.2 模板引擎选型

**现状**：`String.replace('{{KEY}}', val)` + `new RegExp('\\{\\{' + key + '\\}\\}', 'g')`。

**问题**：
- 不支持条件渲染（如「有 benchmarks 才渲染该 section」需在 JS 里拼字符串）；
- 不支持循环（如渲染 94 个数据集需在 JS 里 for 循环拼 HTML）；
- 转义需手动调 `esc()`，易遗漏。

**选型对比**

| 方案 | 体积 | 学习成本 | 是否支持条件/循环 | 推荐度 |
|------|------|---------|------------------|--------|
| 现状 String.replace | 0 | 0 | ❌ | - |
| Mustache.js | 2.4KB | 低 | ✅ | ⭐⭐⭐⭐ |
| Handlebars | 21KB | 中 | ✅ + Helper | ⭐⭐⭐ |
| EJS | 1.5KB | 低 | ✅ | ⭐⭐⭐ |
| 自研 50 行模板引擎 | 1KB | 0 | ✅ 基础 | ⭐⭐⭐⭐⭐ |

**推荐**：自研极简模板引擎（约 50 行），支持 `{{var}}`、`{{#if cond}}...{{/if}}`、`{{#each list}}...{{/each}}`、`{{esc var}}`。

**示例实现**

```javascript
// build/templates/render.js
function render(tpl, data) {
  // {{#each list}}{{this.name}}{{/each}}
  tpl = tpl.replace(/\{\{#each (\w+)\}\}([\s\S]*?)\{\{\/each\}\}/g, (_, key, body) =>
    (data[key] || []).map(item => render(body, item)).join('')
  );
  // {{#if cond}}...{{/if}}
  tpl = tpl.replace(/\{\{#if (\w+)\}\}([\s\S]*?)\{\{\/if\}\}/g, (_, key, body) =>
    data[key] ? body : ''
  );
  // {{esc var}} — 转义
  tpl = tpl.replace(/\{\{esc (\w+)\}\}/g, (_, key) => esc(data[key] || ''));
  // {{var}} — 原样输出
  tpl = tpl.replace(/\{\{(\w+)\}\}/g, (_, key) => data[key] ?? '');
  return tpl;
}
```

**收益**：详情页构建函数可从 250 行降到 80 行，HTML 结构在模板里一目了然。

### 1.3 数据层演进

**现状问题**

中英文双文件（`datasets.json` + `datasets.en.json`）存在：
- 同步靠人工，`validate.cjs` 只能检查 ID 一致性，不能保证字段值同步；
- 94 条 × 20 字段 × 2 语言 = 3760 个值需维护；
- 已出现过 `year` 字段中英不一致的 warning。

**方案 A：合并为单文件 + i18n 字段**

```jsonc
{
  "id": "open-x-embodiment",
  "name": {
    "zh": "Open X-Embodiment",
    "en": "Open X-Embodiment"
  },
  "description": {
    "zh": "Open X-Embodiment 是由 **Google DeepMind** 联合...",
    "en": "Open X-Embodiment is a cross-embodiment..."
  },
  "robotType": ["多机型"],  // 受控词汇，中英通过 build/i18n 映射
  "year": 2024              // 数字，无需翻译
}
```

**优点**：单一数据源，不可能不同步；受控词汇（robotType/task）在构建时通过 i18n 字典翻译。

**缺点**：JSON 体积增大约 30%（key 重复）；手写时需为每个字符串字段写 `{"zh":"","en":""}`，繁琐。

**方案 B：保持双文件 + 自动同步工具**

写一个 `scripts/sync-i18n.cjs`：
- 以中文为主文件；
- 英文文件只填翻译字段（name/description/notes/summary）；
- 其他字段（id/year/robotType/links）自动从中文同步；
- CI 中强制校验，非翻译字段不一致则报错。

**推荐**：**方案 B**。改动小，兼容现有贡献流程，且翻译字段明确。

### 1.4 前端组件化

**现状**：`datasets.html` / `standards.html` / `tools.html` 三个列表页各内嵌 300+ 行原生 JS，逻辑高度重复（筛选 / 分页 / 视图切换 / 对比）。

**目标**：抽取公共 `filter.js`，三个页面复用。

**方案**

```javascript
// src/assets/filter.js（构建时内联到页面）
function initFilter({ data, filters, render, pageSize = 15 }) {
  const state = { search: '', active: {}, page: 1, sort: 'year', view: 'table' };

  function update() {
    const filtered = data.filter(d => matchFilters(d, state));
    const sorted = sortData(filtered, state.sort);
    const paged = paginate(sorted, state.page, pageSize);
    render(paged, state);
  }

  // 绑定 UI 事件
  bindSearchInput(state, update);
  bindFilterChips(state, update);
  bindSortSelect(state, update);
  bindViewToggle(state, update);
  bindPagination(state, update);
}
```

**收益**：三个列表页 JS 从 300 行降到 50 行（只定义 `filters` 配置和 `render` 函数）。

### 1.5 数据集对比功能增强

**现状**：对比结果存在内存，刷新丢失；最多 4 个；无持久化。

**建议**：
- 用 `localStorage` 持久化选中数据集 ID 列表；
- 对比结果生成独立 URL（如 `/compare/?ids=a,b,c`），可分享；
- 对比维度增加「训练成本估算」（基于数据规模 + 模型大小）；

## 2. 性能优化

### 2.1 构建产物体积分析

**当前体积**（估算）

| 产物 | 体积 | 说明 |
|------|------|------|
| 中文站 HTML | ~3.5MB | 94 详情页 × 30KB + 列表页 + 其他 |
| 英文站 HTML | ~3.5MB | 同上 |
| `embeddings.json` | ~300KB | 73 实体 × 2048 维 × 4 字节 |
| `embeddings_papers.json` | ~8MB | 1220 chunk × 2048 维 × 4 字节 |
| 静态资源 | ~500KB | Logo PNG + 字体缓存 |
| **总计** | **~16MB** | |

**优化项**

| 优化 | 预期减少 | 实施成本 |
|------|---------|---------|
| HTML 压缩（去注释 + 空白） | -15% (~1MB) | 低，加一个 `html-minifier` 步骤 |
| CSS 外置 + 浏览器缓存 | -30KB × 页面数（首屏） | 中，需改 `head.html` 注入 `<link>` |
| `embeddings.json` 量化到 int8 | -75% (~225KB) | 中，需改 `embed.cjs` + `fc/index.js` |
| `embeddings_papers.json` 按需加载 | -8MB（首屏不加载） | 低，FC 端 lazy load |
| 字体 `font-display: swap` | 首屏文字提前 200ms | 低，改 `<link>` |
| 详情页相关推荐预渲染精简 | -2KB / 页 | 低 |

**优先级**：CSS 外置 > HTML 压缩 > 字体优化 > 向量量化。

### 2.2 搜索后端冷启动优化

**现状**：阿里云 FC 冷启动时加载 `embeddings.json`（300KB）+ `embeddings_papers.json`（8MB），首次请求 2-5s。

**方案对比**

| 方案 | 冷启动时间 | 月成本 | 实施复杂度 |
|------|-----------|--------|-----------|
| 现状 | 2-5s | ~¥50（按量） | - |
| FC 预留实例 | <100ms | ~¥200/月 | 低，控制台配置 |
| 向量数据库（Milvus） | <50ms | ~¥500/月（托管） | 高，需迁移 |
| 阿里云 OpenSearch 向量版 | <50ms | ~¥300/月 | 中，有 SDK |
| 论文向量 lazy load | 1s（仅数据集） | ~¥50 | 低 |

**推荐路径**：
1. **短期（1 周）**：论文向量 lazy load，首屏只加载数据集向量；
2. **中期（1 月）**：FC 预留实例 1 个，消除冷启动；
3. **长期（3 月）**：迁移到阿里云 OpenSearch 向量版，支持 Top-K 加速 + 更大规模。

### 2.3 嵌入模型升级评估

**当前**：百炼 `text-embedding-v4`，2048 维，按量计费。

**候选模型对比**

| 模型 | 维度 | 部署方式 | 成本 | 中文效果 | 推荐度 |
|------|------|---------|------|---------|--------|
| 百炼 text-embedding-v4 | 2048 | API | ¥0.0007/千token | 优秀 | ⭐⭐⭐⭐（现状） |
| Qwen3-Embedding-8B | 4096 | 本地 GPU | 服务器成本 | 优秀 | ⭐⭐⭐⭐ |
| BGE-M3 | 1024 | 本地 CPU 可跑 | 免费 | 良好 | ⭐⭐⭐⭐⭐ |
| OpenAI text-embedding-3-large | 3072 | API | $0.13/百万token | 良好 | ⭐⭐ |
| Cohere embed-v3 | 1024 | API | $0.10/百万token | 一般 | ⭐⭐ |

**推荐**：评估 **BGE-M3**（开源，1024 维，CPU 可跑，中文效果良好）。迁移后：
- 向量体积减少 50%（1024 vs 2048）；
- 无 API 成本；
- 可本地部署，无网络延迟。

**迁移成本**：改 `embed.cjs` 的 model 参数 + 重建向量索引 + 改 `fc/index.js` 的 `cosineSim`（维度参数化）。约 1 天工作量。

### 2.4 CDN 与缓存优化

**现状**：GitHub Pages 自带 CDN，但无缓存控制头。

**建议**：
- HTML 文件：`Cache-Control: max-age=300`（5 分钟，平衡更新与缓存）；
- 静态资源（PNG/CSS/JS）：`Cache-Control: max-age=31536000, immutable`（1 年，文件名带 hash）；
- 由于 GitHub Pages 不支持自定义 header，需在 HTML 中加 `<meta http-equiv="Cache-Control">` 或迁移到 Cloudflare Pages。

**Cloudflare Pages 迁移收益**：
- 自定义缓存规则；
- 全球 200+ 节点（GitHub Pages 节点较少）；
- 免费 SSL + DDoS 防护；
- Workers 可做边缘 AI 路由。

**迁移成本**：低，只需改 `deploy.yml` 的部署目标。

## 3. 数据治理优化

### 3.1 自动化校验闭环

**现状**：`validate.cjs` 和 `scan-fiction-risk.py` 需手动运行，CI 只检查模板渲染和数量。

**目标 CI 流程**

```yaml
# .github/workflows/deploy.yml 新增
- name: Data validation
  run: |
    node scripts/validate.cjs
    python3 scripts/scan-fiction-risk.py --threshold=5 --format=json > fiction-risk.json

- name: Check fiction risk
  run: |
    HIGH_RISK_COUNT=$(jq '.high_risk | length' fiction-risk.json)
    if [ "$HIGH_RISK_COUNT" -gt 0 ]; then
      echo "::warning::${HIGH_RISK_COUNT} high-risk datasets detected"
      # 不 fail，但会在 PR 评论里提醒
    fi

- name: Comment PR with risk report
  if: github.event_name == 'pull_request'
  uses: actions/github-script@v7
  with:
    script: |
      const report = require('./fiction-risk.json');
      const body = `## 🔍 AI Fiction Risk Report
      - 🔴 High risk: ${report.high_risk.length}
      - 🟡 Medium risk: ${report.medium_risk.length}
      ${report.high_risk.slice(0, 5).map(r => `- **${r.name}** (${r.score}分): ${r.flags.join(', ')}`).join('\n')}
      `;
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        body
      });
```

**收益**：每个 PR 自动跑虚构扫描，高风险条目自动评论提醒，强制人工复核。

### 3.2 链接健康监控

**现状**：`validate.cjs --links` 需手动跑，约 30s。

**建议**：新增 `.github/workflows/link-check.yml`，每周一 9:00 自动跑：

```yaml
on:
  schedule:
    - cron: '0 1 * * 1'  # 北京时间周一 9:00
jobs:
  link-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: node scripts/validate.cjs --links
      - name: Create issue for dead links
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: '🔗 死链检测报告 ' + new Date().toISOString().slice(0,10),
              body: require('fs').readFileSync('link-report.md', 'utf8'),
              labels: ['maintenance', 'dead-link']
            });
```

### 3.3 数据贡献流程升级

**现状**：提交数据集靠邮件或 PR 手动编辑 JSON。

**目标**：Web 表单提交 → 自动生成 PR → CI 校验 → 合并。

**实现方案**

1. `/submit/` 页面增加「在线提交表单」（除现有邮件 + GitHub 方式）；
2. 表单字段对应 `datasets.json` 的 schema，内置校验；
3. 提交后调用 GitHub API 创建 PR：

```javascript
// fc/submit.js（新函数）
async function createDatasetPR(payload) {
  const newDataset = sanitize(payload);
  const branch = `add-${newDataset.id}`;
  const content = JSON.stringify(newDataset, null, 2);

  // 1. Fork 仓库（或用 bot 账号）
  // 2. 创建分支
  // 3. 修改 docs/data/datasets.json（追加到数组）
  // 4. 提交 commit
  // 5. 创建 PR
  // 6. PR 描述自动填充数据集信息
}
```

4. PR 自动触发 CI 跑 `validate.cjs` + `scan-fiction-risk.py`；
5. 高风险条目自动评论，要求人工复核；
6. 合并后自动重建向量索引 + 部署。

**收益**：降低贡献门槛，从「编辑 JSON + 提 PR」降到「填表单」。

### 3.4 数据质量评分体系

**现状**：`scan-fiction-risk.py` 输出风险分，但未展示给用户。

**建议**：在数据集详情页展示「数据完整度评分」（0-100），基于：
- 字段完整度（40%）：必填字段是否齐全；
- 链接有效性（20%）：official/paper/github/huggingface 是否可访问；
- 描述丰富度（20%）：description 字数 + 是否有 usage；
- 时效性（10%）：changelog 条数 + 最近更新；
- 引用完整性（10%）：是否有 BibTeX。

**展示**：详情页顶部加 badge「📊 数据完整度 85/100」，点击展开评分明细。

**收益**：用户快速判断数据集信息可信度；贡献者有动力补全字段。

## 4. 用户体验优化

### 4.1 搜索体验升级

#### 4.1.1 查询意图识别

**现状**：所有 query 走同一检索流程（embedding + 关键词加权）。

**建议**：识别查询意图，走不同策略。

| 意图 | 示例 query | 策略 |
|------|-----------|------|
| 对比 | "OpenVLA vs RT-2" | 识别实体名，返回两个数据集详情 + 对比表 |
| 推荐 | "人形机器人双臂操作" | 走反向推荐引擎（6 维加权） |
| 下载 | "DROID 下载" | 直接返回数据集详情 + 下载链接高亮 |
| 方法论 | "π0 用什么架构" | 走论文知识库检索（+10 boost） |
| 列举 | "有哪些开源数据集" | 走标准检索，返回 Top 20 |

**实现**：在 `fc/index.js` 加意图分类器（规则 + 关键词匹配即可，无需 ML）：

```javascript
function classifyIntent(query) {
  if (/vs|对比|比较/.test(query)) return 'compare';
  if (/下载|download|获取/.test(query)) return 'download';
  if (/架构|方法|原理|怎么/.test(query)) return 'methodology';
  if (/哪些|列表|全部/.test(query)) return 'list';
  return 'recommend';  // 默认
}
```

#### 4.1.2 LLM 流式输出

**现状**：`/api/assistant` 等待 DeepSeek 完整生成后返回，首字延迟 3-8s。

**建议**：改用 SSE（Server-Sent Events）流式返回。

**FC 端改造**：

```javascript
// fc/index.js
if (path === '/api/assistant/stream') {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    ...corsHeaders,
  });

  const stream = await fetch('https://api.deepseek.com/v1/chat/completions', {
    method: 'POST',
    headers: { 'Authorization': `Bearer ${apiKey}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ model, messages, stream: true, ... }),
  });

  for await (const chunk of stream.body) {
    res.write(`data: ${chunk}\n\n`);
  }
  res.end();
}
```

**前端改造**：用 `EventSource` 或 `fetch + ReadableStream` 接收，逐字渲染。

## 5. 安全优化

### 5.1 API 防滥用

**现状**：`/api/search` 和 `/api/assistant` 无任何鉴权，硬编码 FC 地址可被滥用。

**风险**：
- DeepSeek API 成本：每次 assistant 调用约 ¥0.05，被刷 1 万次 = ¥500；
- 百炼 embedding：每次 ¥0.001，被刷影响较小；
- DDoS 风险。

**方案**

| 层级 | 方案 | 成本 | 效果 |
|------|------|------|------|
| L1 | Referer / Origin 白名单 | 免费 | 防简单盗用 |
| L2 | 简单速率限制（每 IP 30 次/分钟） | 低 | 防刷 |
| L3 | API Key（前端带 key） | 中 | 防盗用 |
| L4 | 用户登录 + JWT | 高 | 强鉴权 |

**推荐**：L1 + L2 起步，后续按需升级。

**L2 实现**（FC 端内存限流）：

```javascript
const rateLimit = new Map(); // ip -> { count, resetAt }
function checkRate(ip) {
  const now = Date.now();
  const r = rateLimit.get(ip) || { count: 0, resetAt: now + 60000 };
  if (now > r.resetAt) { r.count = 0; r.resetAt = now + 60000; }
  r.count++;
  rateLimit.set(ip, r);
  return r.count <= 30;
}
```

### 5.2 XSS 防护

**现状**：
- 用户输入（搜索 query）通过 `esc()` 转义，安全；
- 博客 `contentHtml` 直接注入 HTML（信任作者），有风险；
- LLM 返回的 `reply` 通过 `simpleMarkdown` 渲染，未做完整 XSS 清洗。

**建议**：
- 博客内容构建时用 DOMPurify 清洗（Node 端用 `jsdom` + `dompurify`）；
- LLM 返回内容前端用 DOMPurify 清洗后再渲染；
- 引入 CSP（Content Security Policy）header：

```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self' https://superdaa-search-kuccdqlnpa.cn-hangzhou.fcapp.run;
```

### 5.3 内容安全

**风险**：用户通过 giscus 评论注入恶意链接 / 内容。

**现状**：giscus 基于 GitHub Discussions，GitHub 有内容审核，风险低。

**建议**：保持现状，定期检查 Discussions。

## 附录：关键决策点

### 决策 1：是否修改 NC 许可证？

**选项 A**：保持 CC BY-NC-SA 4.0
- 优点：保护数据不被商业滥用；
- 缺点：限制 B 端变现。

**选项 B**：改为 CC BY-SA 4.0（允许商业）
- 优点：B 端可自由使用，降低销售阻力；
- 缺点：竞品可免费使用数据。

**选项 C**：双许可（非商业免费 + 商业授权付费）
- 优点：兼顾开源 + 商业；
- 缺点：法律复杂度高。

**推荐**：**选项 C**。数据保持 NC，但向付费企业客户提供商业授权（含在 SaaS / API 套餐中）。

### 决策 2：是否接受投资？

**选项 A**：保持自力更生
- 优点：完全控制权；
- 缺点：增长慢，可能错过窗口期。

**选项 B**：接受产业投资（机器人公司战投）
- 优点：资源协同 + 客户导入；
- 缺点：可能影响中立性。

**选项 C**：接受财务投资（VC）
- 优点：纯资金支持，不干预业务；
- 缺点：增长压力大。

**推荐**：**Y2 阶段选项 B**（产业投资），**Y3 阶段选项 C**（VC 跟投）。

### 决策 3：是否自建数据集交易平台？

**选项 A**：只做信息导航
- 优点：轻资产，无合规风险；
- 缺点：天花板低。

**选项 B**：自建交易平台
- 优点：收入天花板高；
- 缺点：重资产，合规复杂。

**选项 C**：与现有平台合作（HuggingFace / ModelScope）
- 优点：轻资产 + 借力；
- 缺点：依赖合作方。

**推荐**：**Y3 阶段选项 C**，先合作验证模式，长期考虑选项 B。
